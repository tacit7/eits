# Agent Schedules Design Review

## Overall assessment

The design is serviceable, but several parts are weaker than they look. The main problems are data integrity, unclear semantics, and a mismatch between what the UI implies and what the storage model allows.

## What looks solid

- Reusing the existing Jobs LiveView instead of adding new routes or modules is a reasonable first move.
- Storing a snapshot of `instructions` in the scheduled job config is the right choice if scheduled runs should remain stable after prompt edits.
- Using CSS breakpoints for modal vs drawer is fine; no JS viewport detection is needed.
- Making `prompt_id` optional and backward-compatible is a practical migration path.

## What is off

### 1. This is not actually a database join

The design says the tab "joins against `scheduled_jobs`", but the implementation described is:

- load prompts
- load matching jobs
- build a `prompt_id -> job` map in LiveView

That is not a SQL join. It is an application-side association map.

### 2. UI-only duplicate prevention is not enough

The design says duplicate prevention is not needed at the DB level because the button changes from “+ Schedule” to “Edit”.

That is not strong enough.

You can still get duplicates from:

- double submits
- race conditions
- multiple browser tabs
- future API endpoints
- admin scripts
- bugs in LiveView state

Right now the design implies one schedule per prompt, but the schema still allows many. That mismatch should be resolved explicitly.

You need to decide whether the real rule is:

- one schedule per prompt total
- one schedule per prompt per project
- multiple schedules per prompt are allowed

If the rule is one schedule per prompt, enforce it below the UI.

### 3. `prompt_id` in JSONB is okay for v1, but weak for a core relationship

Querying like this works:

```sql
config::jsonb->>'prompt_id'
```

But if `prompt_id` defines the relationship between prompts and scheduled jobs, it is becoming first-class data. First-class data should usually be stored in a real column.

Keeping it only in JSONB makes these things worse:

- constraints
- indexing
- reporting
- joins
- query clarity

A better design would add a nullable `prompt_id` column on `scheduled_jobs`, while still keeping runtime execution payload in `config`.

### 4. "All active prompts" may be the wrong dataset

Loading all active prompts on mount might be fine for a small system, but the design does not answer basic scoping questions:

- are prompts global or project-scoped?
- in `ProjectLive.Jobs`, should only that project’s prompts be shown?
- what does "active" mean; enabled, not archived, not soft-deleted?
- will prompts remain small enough that full-load on mount stays acceptable?

In `ProjectLive.Jobs`, showing all prompts sounds especially questionable unless that is intentional.

### 5. Project override behavior is underspecified

The design mentions an optional project override dropdown and then stores `project_path`, but it does not define resolution order.

That needs to be explicit. A sane order would be:

1. explicit project override from the form
2. prompt default project, if any
3. current page context project
4. validation error if none exists

Without that, different code paths will make different assumptions.

### 6. Model selection changes the meaning of “schedule this prompt”

The form allows selecting a model while also copying instructions from the prompt. That means the resulting scheduled job is not a live reference to a prompt; it is a snapshot seeded from the prompt.

That is probably the right decision, but the design should name it clearly.

Current behavior is effectively:

> create a scheduled agent job using this prompt as the seed

It is not:

> create a live schedule that always follows the current prompt definition

That distinction matters for user expectations and future edits.

### 7. Deleted prompt behavior is operationally fine, but UX is incomplete

The design says a job should keep running after prompt deletion because the instructions were copied into config.

That part is fine.

But then what happens in the Agent Schedules tab if the prompt is gone?

If the tab is prompt-driven, the deleted prompt disappears and the job may become invisible there even though it still exists and still runs.

That is an operational gap.

Possible fixes:

- show orphaned scheduled agent jobs in a separate section
- prevent deleting prompts that still have schedules
- soft-delete prompts and keep them visible with a deleted badge
- show a “prompt deleted” state in the UI

### 8. The tab is not really just another jobs tab

“All Jobs” is job-centric.

“Agent Schedules” as designed is prompt-centric with schedule state layered on top.

That is not necessarily bad, but it means this is not simply another filtered jobs view. It is really a prompt management surface embedded inside the Jobs LiveView.

That should be acknowledged explicitly so the implementation does not get twisted later trying to pretend everything is the same resource type.

## What I would change

### Minimum acceptable version

Keep the current overall shape, but tighten the weak points:

- keep it inside the existing Jobs LiveView
- keep snapshot semantics for instructions
- keep the modal/drawer setup
- enforce uniqueness in the backend if one schedule per prompt is the intended rule
- define project resolution order clearly
- handle orphaned jobs visibly in the UI
- stop describing the app-side lookup as a join

### Better version

If you want cleaner architecture:

- add nullable `scheduled_jobs.prompt_id`
- possibly add nullable `scheduled_jobs.project_id`
- add indexes
- enforce uniqueness according to the real business rule
- query with actual joins or preloads
- render prompt rows with associated schedule state directly from proper relational data

That would be a more standard and maintainable design.

## Suggested rewrite of the architecture summary

> Add an “Agent Schedules” tab inside the existing Jobs LiveView. The tab loads active subagent prompts and existing `spawn_agent` scheduled jobs, then builds a prompt-to-job lookup for rendering schedule state per prompt. Creating a schedule generates a `spawn_agent` job using a snapshot of the prompt instructions, selected model, and resolved project path. Scheduled jobs store an optional `prompt_id` for traceability.

## Bottom line

The design is workable, but these issues should be addressed before calling it clean:

- duplicate prevention should not rely only on the UI
- `prompt_id` in JSONB is acceptable for v1, but weak for a core relationship
- orphaned jobs need a visibility story
- project scoping and project resolution need explicit rules
- the tab is prompt-centric, not just another jobs list
- the current implementation is not actually a database join
