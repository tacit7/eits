# EITS Assistant Architecture; Refined Response

Claude’s last correction about the **tool registry living in the DB** is right. The earlier suggestion was too loose.

If tool access varies by:

- assistant
- project
- team
- approval level

then static config stops being “simple” and starts being a hidden policy engine smeared across code. That is a bad design. Policy wants to be queryable, inspectable, and editable without redeploying.

## What now looks correct

### `assistants`
This shape is good:

```text
assistants
  id
  name
  prompt_id
  model
  effort
  tool_policy
  trigger_type
  project_id
  team_id
  active
  inserted_at
  updated_at
```

### `sessions`
And this extension also makes sense:

```text
sessions
  assistant_id
  trigger_type
  run_context
```

That preserves `sessions` as the single runtime record while allowing assistant-backed executions. Clean, minimal, and grounded in your actual schema.

## Refined take

### 1. `trigger_type` on both `assistants` and `sessions` is valid
But they mean different things, so name them carefully.

Right now you have:

- `assistants.trigger_type`
- `sessions.trigger_type`

That is not automatically wrong, but it can get muddy fast.

#### Better interpretation
- **`assistants.trigger_type`** = allowed/default invocation mode
- **`sessions.trigger_type`** = actual invocation source for this run

Those are different concepts.

So either:

#### Option A; keep both, but rename one
- `assistants.default_trigger_type`
- `sessions.trigger_type`

#### Option B; make assistant field plural or policy-based
- `assistants.trigger_policy` JSONB
- `sessions.trigger_type`

Option A is better for v1 because it is simpler and obvious.

Example:
- assistant default trigger = `task_dispatch`
- actual session trigger = `manual`

That tells a coherent story.

### 2. `tool_policy` JSONB is fine for v1
This is the right tradeoff.

You do **not** need a giant normalized permission model yet. JSONB is enough if you keep the shape disciplined.

Example:

```json
{
  "allowed": ["read_file", "search_tasks", "spawn_subagent"],
  "requires_approval": ["spawn_subagent", "write_file", "run_command"],
  "denied": ["delete_project"]
}
```

That works.

But the important part is this: the **tool catalog itself** should still live in a table.

So the clean split is:

- **tools table** = what tools exist and their metadata
- **assistant.tool_policy** = which of those tools this assistant can use and under what conditions

That is the right hybrid.

## Recommended thin tool schema

You do not need to overbuild this. Just enough structure to stop future pain.

### `tools`
```text
tools
id
name
description
destructive
requires_approval_default
active
inserted_at
updated_at
```

Optional later:
- `input_schema`
- `scope_type`
- `handler_module`
- `timeout_ms`

For now, this is enough.

Then `assistants.tool_policy` can reference tool names or ids. Using **tool names** in JSONB initially is better because they are easier to read and debug.

### 3. `run_context` is a good idea
This one is useful, but only if you keep it constrained.

You listed:
- goal
- task_id
- source info

That’s good.

What you do **not** want is `run_context` becoming a junk drawer for random state because the schema felt inconvenient that day.

#### Good `run_context` contents
- `goal`
- `task_id`
- `scheduled_job_id`
- `dispatched_by`
- `source_channel_id`
- `source_note_id`
- `source_event`

#### Bad `run_context` contents
- full prompt body
- large message history
- mutable execution state
- anything that should really live in a first-class column

Use it for invocation metadata, not as a second database hidden inside JSON.

## One thing I’d still tighten

### `effort` and `model`
These are fine as plain text for now, but be careful.

If `effort` is tied to Claude CLI semantics and `model` is also provider-specific, then your assistant definition is implicitly runtime-coupled. That may be okay for internal use, but name them honestly.

If this is Claude-only right now, consider:

- `model`
- `reasoning_effort` or `execution_effort`

instead of generic `effort`, because `effort` by itself is vague and will age badly.

## How to implement this in EITS

### Step 1; assistants wraps prompts
Create assistant CRUD on top of prompts.

Assistant creation flow:
- select existing prompt
- choose model
- choose reasoning effort
- choose allowed tools
- choose which tools require approval
- optional project/team scoping
- choose default trigger type

That gives you reusable execution profiles without duplicating prompt content.

### Step 2; session spawn from assistant
Update the spawn flow so user can choose:
- ad hoc session
- assistant-backed session

When spawning from assistant:
- set `sessions.assistant_id`
- copy `trigger_type`
- populate `run_context`
- resolve prompt content from `prompt_id`
- apply tool policy

That is where the feature becomes real.

### Step 3; tool enforcement before tool execution
When a session tries to invoke a tool:

1. resolve session assistant
2. read assistant tool policy
3. check tool exists and is active
4. check if allowed
5. check if approval required
6. either enqueue approval or execute

This logic should live in one place. Something like:

- `Eits.Assistants.ToolPolicy`
- `Eits.Assistants.ToolExecutor`

Do not scatter permission checks across random workers and LiveViews.

### Step 4; approval inbox
Before broad execution tools, build the approval pipeline.

Minimum approval record could be:

### `tool_approvals`
```text
id
session_id
assistant_id
tool_name
payload jsonb
status              -- pending, approved, denied, expired
requested_by_type
requested_by_id
reviewed_by_id
reviewed_at
inserted_at
updated_at
```

Even if the tool call log later grows into its own thing, this gives you a proper checkpoint now.

### Step 5; scoped notes as memory
Still the right v1 move.

Use notes for:
- session summaries
- project conventions
- assistant operating notes
- decision logs

You can later formalize memory if the retrieval/query patterns become painful.

## One more terminology fix

Still strongly agree with narrowing `Agent`.

From what you quoted:

> Agent has persona_id, description, parent_agent_id, session_id all crammed together.

That is exactly the kind of model that starts convenient and ends up semantically dishonest.

It is mixing:
- reusable identity
- lineage
- runtime linkage
- description metadata

Eventually you’ll want to split that, but not necessarily now.

For this assistant feature, at minimum:
- let `Assistant` become the reusable config concept
- let `Session` remain runtime
- avoid adding more meaning to `Agent`

Do not keep feeding the overloaded term.

## Final judgment

This is the version worth backing now:

- **Keep `sessions` as runtime source of truth**
- **Add thin `assistants` wrapping `prompts`**
- **Add a real `tools` table**
- **Use `assistant.tool_policy` JSONB for per-assistant permissions**
- **Add `assistant_id`, `trigger_type`, and `run_context` to sessions**
- **Use notes for v1 memory**
- **Build approval before broad tool execution**

That is a solid internal-tool architecture. Not overbuilt, not sloppy, and not pretending every future requirement needs to be solved today.
