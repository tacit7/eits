# Review: Note Full-Page Editor Design Spec

## Overall

This spec is mostly solid. The architecture is sane, the separation from the inline editor is the right call, and the flow is easy to follow. It is implementation-ready in broad strokes, but a few parts are either inaccurate, underspecified, or likely to create annoying edge-case bugs.

## What looks good

- A dedicated full-page editor instead of stretching the inline editor into something it was not built for.
- `return_to` validation exists and is scoped to known internal paths.
- The full-page hook is independent from the inline hook. That is the right boundary.
- Client-side status bar updates for line and column are the right choice.
- Save semantics are simple and clear; save stays on page and shows a transient success state.
- `current_path` threading is explicit and easy to reason about.

## What is off

### 1. The file paths look wrong

The spec uses paths like:

- `lib/eye_in_the_sky_web_web/live/note_live/edit.ex`
- `lib/eye_in_the_sky_web_web/components/notes_list.ex`
- `lib/eye_in_the_sky_web_web/router.ex`

That `eye_in_the_sky_web_web` segment looks wrong unless the app structure is already broken in a very specific way. In a normal Phoenix app I would expect something like:

- `lib/eye_in_the_sky_web/live/note_live/edit.ex`
- `lib/eye_in_the_sky_web/components/notes_list.ex`
- `lib/eye_in_the_sky_web/router.ex`

This is probably just a typo in the spec, but it should be fixed before someone copies it directly into the codebase.

### 2. `handle_params/3` needs explicit reset behavior

The spec says:

- `mount/3` assigns defaults
- `handle_params/3` loads the note by `:id`

That is fine, but it should explicitly say what happens if the same LiveView instance receives a different note ID later.

At minimum, when `handle_params/3` loads a different note, it should reset transient state such as:

- `saved: false`

Otherwise it is easy to end up with stale UI state, like opening a second note and still seeing the prior note’s green “Saved ✓” state.

### 3. `update_title` is underspecified

The title update path is weaker than the body save path. The spec currently says:

```elixir
Notes.update_note(socket.assigns.note, %{title: title})
```

on blur.

Missing details:

- Is the title trimmed before saving?
- Are blank titles allowed?
- What happens on validation failure?
- Does the input keep the typed value or revert to persisted value?
- Does a successful update replace `@note` with the returned struct?

The spec should define the success and failure behavior clearly.

### 4. `note_saved` only defines the happy path

The spec says body save should:

- update the note
- set `saved: true`
- schedule `:clear_saved` after 3 seconds

It does not define failure behavior.

It should say what happens if `Notes.update_note/2` fails:

- do not set `saved: true`
- show flash or inline error
- preserve editor contents client-side
- do not overwrite `@note.body` unless the save actually succeeds

That is the main interaction path; it should not have a hole in the spec.

### 5. The `:clear_saved` timer can race

This is a real bug risk.

Example:

1. user saves
2. timer A is scheduled for 3 seconds
3. user saves again 1 second later
4. timer B is scheduled for 3 seconds
5. timer A fires early relative to the second save and clears the saved state too soon

The spec should account for this by either:

- storing and canceling the previous timer reference, or
- using a token/version check so only the latest timer clears the state

Without that, the “Saved ✓” duration is not reliably tied to the most recent save.

### 6. `window.location.href` is probably the wrong navigation primitive

The hook spec says Escape should do:

```javascript
window.location.href = this.el.dataset.returnTo
```

That works, but it causes full browser navigation. In a Phoenix LiveView app, internal navigation should usually use LiveView-aware navigation unless a full reload is intentional.

Same problem with the header back link using `href={@return_to}`. If the target is another LiveView route in the app, the spec should at least justify why plain browser navigation is acceptable instead of `navigate`.

This is not fatal, but it is less consistent with normal LiveView routing behavior.

### 7. `data-body` needs a note about escaping and payload size

The spec says the note body is passed via `data-body` as plain UTF-8.

That can be fine, but markdown bodies may contain quotes, angle brackets, newlines, and other characters that make attribute transport messy if somebody misunderstands how HEEx escaping works.

The spec should explicitly say the body is rendered through normal HEEx attribute escaping and read back from the DOM by the hook.

Also, if note bodies can get large, using a `data-*` attribute as the transport mechanism may become clunky. Not necessarily wrong for v1, but worth acknowledging.

### 8. “Shares no runtime code” is too rigid

The spec says the full-page hook:

> Shares no runtime code with the existing `NoteEditorHook`.

The architectural point is valid; the full-page editor should not depend on inline-editor behavior. But this wording is too absolute.

A better rule would be:

- no shared stateful behavior
- no inheritance or coupling to inline-editor assumptions
- small stateless shared utilities are okay if they emerge naturally

Otherwise the spec starts forbidding sensible reuse for no real reason.

### 9. “Parent context badge” is not defined

The render layout includes a “parent context badge,” but the spec never explains:

- what it shows
- whether it is clickable
- what it does for project notes vs standalone notes
- what happens when there is no parent context

Right now that is just a vague noun in the header requirements. It needs definition or removal.

### 10. `current_path` is not really “static”

The spec says:

> `current_path` is passed as a static value in each LiveView's `render/1`, not via assigns

That is not accurate for this example:

```elixir
~p"/projects/#{@project.id}/notes"
```

That value is derived during render and depends on assigns. It is not static.

A more correct phrasing would be:

> `current_path` is passed directly from each LiveView render callsite rather than being stored as a dedicated assign in the parent LiveView.

### 11. Missing behavior if the note changes or disappears elsewhere

This is probably out of scope, but the spec should say so explicitly.

Cases not covered:

- note updated elsewhere while this editor is open
- note title updated elsewhere while this page is open
- note deleted while this page is open

The spec does not need collaborative editing support, but it should at least define the basic failure story, for example:

- last write wins
- if note no longer exists, show error and navigate away on next save attempt

### 12. “Keyboard hints” are mentioned too vaguely

The status bar mentions “keyboard hints,” but the spec only clearly defines:

- Save with `⌘S`
- Escape to leave

If those are the only hints shown, say that. If there are more, list them. “Keyboard hints” is too vague as written.

## Recommended changes

### Tighten event behavior

For `note_saved`:

- on success: persist, set `saved: true`, clear existing saved timer if needed, schedule new timer
- on failure: show error state, do not set saved, preserve editor contents

For `update_title`:

- define title normalization, validation, success behavior, and failure behavior

### Fix navigation strategy

Use LiveView-aware navigation for internal routes unless a full reload is intentional and justified.

### Reset transient state on param change

When loading a note in `handle_params/3`, reset transient UI state such as `saved`.

### Clarify the body transport contract

Document that the body is safely escaped by HEEx before being placed in `data-body`, and note that this choice may need reevaluation if note sizes grow significantly.

### Define the header context badge

Specify exactly what the badge shows and whether it is interactive.

### Clean up wording

- fix the file paths
- replace “static value” with more accurate wording
- soften the “shares no runtime code” statement so it allows stateless utility reuse if needed

## Bottom line

This is a good spec. The page boundary is right, the separation from the inline editor is right, and the data flow is understandable.

The main things to fix before implementation are:

1. Correct the file paths.
2. Add explicit failure behavior for title and body saves.
3. Handle the transient save timer safely.
4. Reconsider plain browser navigation for internal routes.
5. Define the parent context badge.
6. Clean up the wording around `current_path` and code sharing.

So the design is not broken. It is just missing a few operational details that will turn into real bugs or inconsistent behavior if nobody pins them down now.
